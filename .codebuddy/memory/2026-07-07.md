## 修复 LocationPage 两处 UI Bug

### Bug 1：位置管理页位置卡片间有间隙
- 现象：位置卡片（位置条）之间有空白间隙。
- 根因：`src/pages/LocationPage.tsx` 中 `locationCard.style.marginBottom: 16`（顶级）和子位置 `marginBottom: 8`（level>0 内联样式）。
- 修复：两者均改为 `marginBottom: 0`。

### Bug 2：「最近移动」误把新建物品当成今天移动
- 现象：用户今天没移动任何物品，但「最近移动」显示了"今天"的条目。
- 根因：`src/database/db.ts` 的 `createItem` 把 `movedAt` 默认设为 `now`（创建时间）；`mapItemRow` 又用 `row.movedAt ?? row.createdAt` 兜底。导致只要新建/添加物品就被算作"移动"。种子数据（seedRichData/seedDemoData）通过 raw SQL 显式设置 movedAt 为若干天前，本身没问题。
- 修复：
  - `createItem`：`movedAt: now` → `movedAt: ''`（新建物品不再冒充移动）。
  - `mapItemRow`：`movedAt: row.movedAt ?? row.createdAt` → `movedAt: row.movedAt ?? ''`。
  - 注意：`getIdleDays`/`ItemsPage`/HomePage 调用处都用 `movedAt || updatedAt` 兜底，改空字符串不影响闲置天数计算。
- 验证：重新 `expo export` + `pm clear com.mobile` + 重启（自动用修复后代码重新填充演示数据），「最近移动」只含演示移动（若干天前），无"今天"。
- 说明：本次 `pm clear` 会清空用户手动添加的数据并重新填充演示数据；若需保留手动数据，勿执行 clear。

## 修复 CropPage 不能自由框选的问题

- 现象：拍照/选图进入裁剪页后，只能移动/拖拽已有裁剪框的句柄，无法在空白处自由拖出新选区（"不能自由框选"）。
- 根因：`CropPage.tsx` 把触摸处理拆到框内 move 区域和各句柄的 `TouchableWithoutFeedback` 上，空白处（图片其他区域）没有任何触摸处理，所以无法绘制新选区。
- 修复方案：将触摸逻辑统一到 `imageContainer`（`TouchableWithoutFeedback` 加 `onTouchStart/Move/End`），新增命中检测 `hitTest(px,py)`：
  - 命中角/边句柄 → 对应 resize 类型；
  - 命中框内 → `move`；
  - 命中图片其余区域 → 新增 `draw` 模式：从按下点拖到当前点形成自由选区（限制在图片区域内）；
  - 图片区域外 → `none`（不响应）。
  - 触摸坐标改用 `event.nativeEvent.locationX/locationY`（相对容器），对 move/resize 的 delta 计算无影响，对 draw 的绝对坐标正确。
  - 自由框选即使未达 MIN_CROP_SIZE 也实时显示，便于反馈；`handleCrop` 增加裁剪区域过小保护。
- i18n：新增 `add.cropTooSmall`（中/英）。
- 验证：`expo export` 生成新 bundle `index-40042ef1...`，重启两次后日志 `isAvailable:false`，新代码已上线。

### 二次修复：TouchableWithoutFeedback 的 onTouchMove 不可靠 → 改用 PanResponder
- 现象：用户实测仍"不能拖拽/自由框选"。TouchableWithoutFeedback 作为容器时 `onTouchMove` 不保证持续触发，导致拖拽/绘制完全失效。
- 修复：把容器 `<View>` 的触摸处理从 `TouchableWithoutFeedback` 的 `onTouchStart/Move/End` 改为标准 **`PanResponder`**（`onStartShouldSetPanResponder`/`onMoveShouldSetPanResponder` 均返回 true；`onPanResponderGrant` 命中检测并初始化 dragRef；`onPanResponderMove` 调用 `updateRect`；`onPanResponderRelease/Terminate` 结束）。
- 为避免 panResponder 闭包拿到过期 cropRect，新增 `cropRectRef` 在每个 render 同步最新值；`hitTest` 改为读取 `cropRectRef.current`（去掉 cropRect 依赖）。
- 坐标仍用 `event.nativeEvent.locationX/locationY`（相对容器，PanResponder 下可靠）。
- 验证：重新 `expo export` 生成新 bundle `index-2a64f038...`，重启两次后日志 `isAvailable:false`，PanResponder 版本上线。**结论：RN 中做拖拽必须用 PanResponder，不要用 Touchable* 的 onTouchMove。**
- 经验补充：OTA 热更新策略为"下载后等待下次 App 重启才生效"，因此每次 expo export 后需 force-stop + 启动 **两次** 才能看到新代码（第一次下载，第二次应用）。

### 三次修复：子 view 吞掉触摸事件 → 全子元素 pointerEvents="none"
- 现象：用户实测"四个定位点无法移动、无法自由框选"，即便逻辑用 PanResponder 也无响应。
- 根因：容器 `<View>` 上接 `panResponder.panHandlers`，但 `imageContainer` 内有 `Image`、4 个暗色遮罩、1 个裁剪框、8 个句柄等 **native 子 view**。这些子 view 默认响应触摸，会**拦截**触摸事件，PanResponder 在容器上根本收不到。
- 修复：把 `imageContainer` 内**所有视觉子元素**都加 `pointerEvents="none"`，让触摸事件穿透到容器，统一由 PanResponder 处理。`onPanResponderGrant` 加一行 `console.log` 便于诊断。
- 验证：重新 `expo export` 生成新 bundle `index-299a8045...`，重启两次后日志 `isAvailable:false`，新代码已上线。
- 经验总结：**RN 中"容器 + panHandlers"模式要求容器内全部子 view 透传触摸**，否则事件被 native 子 view 吃掉。

### 四次修复（真正根因）：createNativeStackNavigator 原生栈下 PanResponder 失效
- 现象：加大量 console.log + pointerEvents="none" 后仍完全无响应；且 logcat 里**完全看不到 CropPage 的日志**（只有 UpdateManager）。
- 关键发现：`expo export` 是**生产模式**，默认 strip 掉 `console.log`（expo 内部 UpdateManager 日志用 I 级被保留，业务代码 console.log 被 babel-preset-expo 移除）→ 之前所有"加日志"都是盲调，等于没诊断。
- **真正根因**：`mobile/App.tsx` 全局包了 `GestureHandlerRootView`，`AppNavigator.tsx` 用 `createNativeStackNavigator`（**原生导航栈**）。在原生栈页面里，React Native 的 `PanResponder` 触摸事件被原生手势系统拦截，JS 端根本收不到 → 四个定位点 + 框选全部失效。前三次修复（Touchable / PanResponder / pointerEvents）都没命中这个根因。
- 修复：把交互层从 `PanResponder` 换成 **`react-native-gesture-handler` 的 `PanGestureHandler`**（基于原生手势识别器，原生栈下能正常工作）。`event.nativeEvent.x/y` 即相对 imageContainer 坐标，等效原 locationX/locationY；`onHandlerStateChange` 用 `State.BEGAN/END/CANCELLED/FAILED` 分发，`onGestureEvent` 处理移动。
- 调试手段升级：生产 bundle 不能靠 console.log，改为在页面顶部加**可见 HUD 浮层**（`pointerEvents:none`）实时显示命中类型/坐标/裁剪框，直接肉眼确认手势生效。日志改 `console.warn`（生产通常保留）。
- 验证：重新 `expo export`（生成新 bundle），重启两次后交付测试；HUD 应显示 ready/start/move/end。
- **关键经验**：① 用 `createNativeStackNavigator` 时，页面内拖拽/手势必须用 `react-native-gesture-handler` 的 Gesture 系列，绝不能用原生 `PanResponder`；② 诊断生产 bundle 不能用 console.log（被剥离），要用 UI 内 HUD 或在 `__DEV__` 下日志；③ 本项目热更新需 force-stop+启动两次。

## 修复 CropPage 裁剪结果与框选不一致

- 现象：裁剪后缩略图与框选区域不匹配，缩略图像被再次裁剪一样。
- 根因：`Image.getSize` 在某些 Android 设备/图片格式上返回的尺寸可能与 `Image` 组件实际渲染尺寸不一致（EXIF 旋转、解码器差异），导致 `scaleX/scaleY` 计算偏差。
- 修复内容 (CropPage.tsx)：
  - 新增 `onImageLoad` 回调：通过 Image 的 onLoad 事件捕获实际加载后原始尺寸，优先级高于 getSize
  - 拆分 `imageOnLoadSizeRef`（优先）和 `imageGetSizeRef`（回退），`getImageSize()` 统一获取
  - `handleCrop` 改用单一 `scale`（imgW/dWidth）替代分开的 scaleX/scaleY，避免不一致
  - 移除 `resizeMode="contain"`（显示尺寸已按宽高比精确计算，contain 可能引入子像素偏移）
  - 新增 `calcDisplayParams` 共用函数，避免 getSize 和 onLoad 路径重复计算
  - 添加 `key` prop 确保图片切换时组件重建
  - 保存 `lastContainerSizeRef` 供 onLoad 重新计算用
  - 添加详细调试日志（console.warn 在生产保留）
